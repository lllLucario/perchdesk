# Surfaces

## Purpose

This file translates the design system into major frontend surfaces so different screens remain consistent without looking identical.

## Marketing / Intro Surfaces

- Use the editorial tone most strongly here
- Larger serif headings, generous whitespace, and chapter-like sections are appropriate
- Green brand moments can be slightly more visible here than in dense app flows

## App Shell

- Navigation, page framing, and top-level layout should feel stable and unobtrusive
- Avoid decorative density in persistent chrome
- The shell should support long use sessions without fatigue

## Dashboard / Home

- Summary information should be calm and scannable
- Avoid turning the page into a metrics board
- Cards should feel like access points, not KPI widgets

## Space Discovery

- This is where the design should feel spatial and inviting
- Use editorial hierarchy to support browsing spaces, buildings, and recommendations
- Availability indicators must remain immediately understandable

## Seat Map

- Function leads here
- Green should indicate selected or available states with precision, not atmosphere
- The surrounding UI can stay soft, but seat status contrasts must remain clear

## Booking Flow

- Confirmation, timing, and rule clarity are the core priorities
- The flow should feel reassuring and low-friction
- Use green most clearly at moments of commitment, success, or selected state

## My Spaces / My Bookings

- These views should feel personal but not overly decorated
- Saved, recent, and recommended content should be easy to differentiate by structure and labeling
- Empty states should teach what the page becomes after use

## Admin Surfaces

- Use the same tokens, but with slightly tighter information density
- Do not drift into a separate “enterprise admin” visual language unless there is a deliberate product reason
